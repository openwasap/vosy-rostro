DECART AI CAMERA - GUÍA TÉCNICA COMPLETA
Sistema de Streaming WebRTC con Filtros de IA en Tiempo Real
PARTE I: VISIÓN GENERAL Y CONTEXTO
1. INTENCIÓN Y PROPÓSITO DEL PROYECTO
1.1 Declaración de Intención
El proyecto Decart AI Camera ha sido concebido con la intención explícita de resolver un problema técnico específico y crítico en el ámbito de las comunicaciones visuales modernas: la necesidad de transmitir video de cámara desde un dispositivo móvil hacia una computadora de escritorio, aplicar filtros de inteligencia artificial en tiempo real sobre dicho flujo de video, y presentar el resultado en una ventana que pueda ser capturada por software de terceros (específicamente SplitCam) para ser retransmitida como cámara virtual hacia aplicaciones de comunicación como WhatsApp, sin depender de soluciones externas como OBS Studio.

La intención fundamental del sistema es proporcionar una solución híbrida que combine la portabilidad de los dispositivos móviles (con sus cámaras de alta calidad) con el poder de procesamiento de las computadoras de escritorio (capaces de ejecutar filtros de IA complejos), todo mediado por una arquitectura WebRTC que permite una comunicación directa peer-to-peer entre ambos dispositivos, minimizando la latencia y maximizando la calidad del video transmitido.

El propósito último es democratizar el acceso a filtros de video avanzados basados en inteligencia artificial para usuarios que desean mejorar su presencia visual en llamadas de video, transmisiones en vivo, o contenido generado para redes sociales, sin necesidad de hardware especializado o conocimientos técnicos avanzados. El sistema está diseñado para ser accesible, intuitivo, y funcional desde el primer momento, con una interfaz de usuario que guía al usuario a través de cada paso del proceso de configuración y uso.

1.2 Contexto Operativo
El contexto operativo del proyecto se sitúa en la intersección de tres tecnologías fundamentales:

WebRTC (Web Real-Time Communication): Una tecnología de código abierto que permite la comunicación en tiempo real entre navegadores y aplicaciones sin necesidad de plugins o software adicional. WebRTC proporciona APIs para acceder a dispositivos de captura de media (cámaras y micrófonos), establecer conexiones peer-to-peer, y transmitir audio y video de manera segura y eficiente. La elección de WebRTC como protocolo de transporte principal se debe a su capacidad de establecer conexiones directas entre pares, lo que minimiza la latencia y reduce la carga en los servidores intermedios.

Inteligencia Artificial Generativa para Video: La integración con la API de Decart AI permite aplicar transformaciones visuales complejas a frames individuales de video, generando efectos como estilos artísticos (anime, pintura al óleo, acuarela), mejoras de calidad, cambios de fondo, y otras modificaciones creativas. Esta tecnología representa el estado del arte en procesamiento de video con IA, utilizando modelos de deep learning entrenados en grandes corpus de datos visuales para producir resultados de alta calidad en tiempo real.

Software de Captura de Pantalla y Cámaras Virtuales: La integración con SplitCam (y potencialmente otros software similares como OBS Virtual Camera, ManyCam, o XSplit VCam) permite que el video procesado por el sistema sea presentado como una fuente de video estándar que cualquier aplicación de comunicación puede utilizar. Esta capa de abstracción es crucial para la usabilidad del sistema, ya que permite que el flujo de video filtrado sea transparente para las aplicaciones finales.

1.3 Objetivos Específicos
El proyecto persigue los siguientes objetivos específicos, cada uno de los cuales ha sido cuidadosamente diseñado para contribuir al propósito general del sistema:

Objetivo 1: Establecimiento de Conexión WebRTC Bidireccional Crear un mecanismo robusto y confiable que permita a un dispositivo móvil (emisor) y una computadora de escritorio (receptor) establecer una conexión WebRTC directa, utilizando un servidor de señalización para coordinar el intercambio inicial de descripciones de sesión (SDP) y candidatos ICE. Este objetivo se cumple a través de una arquitectura de polling HTTP que permite a ambos dispositivos consultar periódicamente el estado de la señalización y obtener los datos necesarios para completar el handshake de WebRTC.

Objetivo 2: Captura y Transmisión de Video Móvil Implementar una interfaz web optimizada para dispositivos móviles que permita acceder a la cámara del dispositivo (frontal o trasera), capturar el flujo de video utilizando la API getUserMedia, y transmitir este flujo a través de la conexión WebRTC hacia el receptor. La interfaz debe ser responsiva, intuitiva, y proporcionar retroalimentación visual clara sobre el estado de la conexión y la calidad del video transmitido.

Objetivo 3: Recepción y Procesamiento de Video en PC Desarrollar una interfaz web para computadora de escritorio que reciba el flujo de video WebRTC, lo presente en un elemento de video HTML5, y proporcione opciones para procesar el video mediante la API de Decart AI. La interfaz debe permitir la visualización tanto del video original como del video filtrado, con controles para alternar entre ambos modos, ajustar la configuración de los filtros, y gestionar la conexión con el servicio de IA.

Objetivo 4: Integración con Decart AI para Filtros en Tiempo Real Establecer una conexión WebSocket con el servicio de Decart AI (endpoint: wss://api3.decart.ai/v1/stream), enviar frames individuales del video para procesamiento, recibir los frames procesados, y renderizarlos en un canvas HTML5. El sistema debe gestionar la backpressure (evitar el envío de nuevos frames mientras se espera la respuesta del anterior), controlar la tasa de frames para optimizar el uso de créditos de la API, y proporcionar indicadores de estado claros sobre la conexión con el servicio de IA.

Objetivo 5: Presentación Fullscreen para Captura por Terceros Diseñar la interfaz del receptor de PC de modo que el video (ya sea original o filtrado) pueda ocupar toda la pantalla, ocultando los controles de la interfaz y proporcionando una experiencia inmersiva que facilita la captura por parte de software de terceros. El modo fullscreen debe ser accesible mediante un botón dedicado y también a través de la API de Fullscreen del navegador.

Objetivo 6: Arquitectura de Señalización Escalable y Robusta Implementar un backend de señalización basado en Express.js que gestione la creación de salas, el almacenamiento de ofertas y respuestas SDP, y el intercambio de candidatos ICE. El sistema debe ser capaz de manejar múltiples salas simultáneas, limpiar automáticamente las salas inactivas, y proporcionar endpoints de salud para monitoreo.

Objetivo 7: Documentación Exhaustiva y Guía de Usuario Producir documentación técnica completa que describa cada aspecto del sistema, desde la arquitectura de alto nivel hasta la implementación de funciones individuales, incluyendo diagramas de flujo, descripciones de protocolos, guías de configuración, y manuales de troubleshooting. Esta documentación debe ser accesible tanto para usuarios finales como para desarrolladores que deseen extender o modificar el sistema.

2. ARQUITECTURA DEL SISTEMA
2.1 Visión General de la Arquitectura
La arquitectura del sistema Decart AI Camera sigue un patrón de tres capas principales, cada una con responsabilidades bien definidas y interfaces claras entre sí. Esta separación de concerns permite un desarrollo paralelo, facilita el testing, y proporciona flexibilidad para futuras extensiones o modificaciones.

Capa 1: Frontend (Cliente Web) La capa de presentación está compuesta por dos aplicaciones web React, una para el dispositivo móvil y otra para la computadora de escritorio. Ambas aplicaciones comparten una base de código común (componentes UI, utilidades, hooks) pero tienen interfaces y comportamientos optimizados para sus respectivos dispositivos. La aplicación móvil se enfoca en la captura de cámara y la transmisión de video, mientras que la aplicación de PC se enfoca en la recepción, procesamiento, y visualización del video.

Capa 2: Backend (Servidor de API) La capa de servidor proporciona los servicios necesarios para la señalización WebRTC y la gestión de configuración. Está implementada como una aplicación Express.js que expone una API RESTful documentada mediante OpenAPI. El servidor no procesa ni almacena el contenido de video (esto ocurre directamente entre los pares WebRTC), sino que se limita a coordinar el establecimiento de la conexión y a mantener la configuración de la integración con Decart AI.

Capa 3: Servicios Externos El sistema se integra con dos servicios externos principales: los servidores STUN de Google (para la resolución de NAT en las conexiones WebRTC) y la API de Decart AI (para el procesamiento de frames con IA). Estas integraciones son cruciales para el funcionamiento del sistema y están diseñadas para ser reemplazables o configurables si en el futuro se desean utilizar servicios alternativos.

2.2 Diagrama de Flujo de Datos
El flujo de datos en el sistema sigue una secuencia bien definida que puede dividirse en cinco fases principales:

Fase 1: Inicialización y Señalización

El usuario abre la página /mobile en su dispositivo móvil y la página / en su PC.
En el PC, el usuario hace clic en "Iniciar recepción", lo que crea una sala de señalización en el backend mediante una petición POST a /api/signaling/rooms.
El backend genera un ID único para la sala y lo devuelve al PC.
El PC comienza a hacer polling periódico al endpoint /api/signaling/rooms/{roomId}/offer para detectar cuando el móvil envíe una oferta.
En el móvil, el usuario hace clic en "Iniciar transmisión", lo que solicita acceso a la cámara mediante getUserMedia.
El móvil crea una conexión WebRTC (RTCPeerConnection), añade los tracks de video y audio, y genera una oferta SDP.
El móvil envía la oferta al backend mediante POST a /api/signaling/rooms/{roomId}/offer.
El móvil también envía sus candidatos ICE recolectados al backend mediante POST a /api/signaling/rooms/{roomId}/ice.
El móvil comienza a hacer polling al endpoint /api/signaling/rooms/{roomId}/answer para detectar la respuesta del PC.
Fase 2: Establecimiento de Conexión WebRTC 10. El PC detecta la oferta del móvil en su polling y la establece como descripción remota (setRemoteDescription).

El PC genera una respuesta SDP (createAnswer) y la establece como descripción local (setLocalDescription).
El PC envía la respuesta al backend mediante POST a /api/signaling/rooms/{roomId}/answer.
El PC también envía sus candidatos ICE al backend.
El móvil detecta la respuesta del PC en su polling y la establece como descripción remota.
Ambos dispositivos intercambian candidatos ICE adicionales a través del backend y los añaden a sus conexiones (addIceCandidate).
La conexión WebRTC peer-to-peer se establece, y el flujo de video comienza a fluir directamente del móvil al PC.
Fase 3: Procesamiento de Video con Decart AI 17. El PC recibe el stream de video en el evento ontrack de la conexión WebRTC y lo asigna a un elemento <video>.

Si el usuario ha configurado una API key de Decart AI, el PC establece una conexión WebSocket con wss://api3.decart.ai/v1/stream.
El PC envía un mensaje de configuración al WebSocket con la API key, el prompt de filtro, y otros parámetros.
El PC inicia un intervalo que captura frames del video (usando un canvas oculto) y los envía al WebSocket como mensajes de tipo "frame".
Decart AI procesa cada frame y devuelve el frame filtrado como mensaje de tipo "frame".
El PC recibe el frame filtrado y lo renderiza en un canvas visible, mostrando el resultado al usuario.
Fase 4: Visualización y Captura 23. El usuario puede alternar entre ver el video original (del elemento <video>) y el video filtrado (del canvas) mediante un botón de toggle.

El usuario puede activar el modo fullscreen para que el video ocupe toda la pantalla, ocultando todos los controles de la interfaz.
El usuario puede utilizar SplitCam (u otro software de captura) para capturar la ventana del navegador y presentarla como una cámara virtual.
Fase 5: Finalización 26. El usuario puede detener la conexión en cualquier momento, lo que cierra la conexión WebRTC, detiene los tracks de media, cierra el WebSocket de Decart AI, limpia los intervalos de polling, y resetea el estado de la interfaz.

2.3 Decisiones de Diseño Arquitectónico
A lo largo del desarrollo del sistema, se tomaron varias decisiones de diseño que merecen explicación detallada, ya que afectan significativamente el comportamiento, rendimiento, y mantenibilidad del sistema.

Decisión 1: Uso de Polling HTTP en lugar de WebSocket para Señalización Aunque WebSocket es la tecnología estándar para señalización en tiempo real en aplicaciones WebRTC, se optó por utilizar polling HTTP para el intercambio de señalización. Esta decisión se basó en varias consideraciones: la simplicidad de implementación (no requiere mantener conexiones persistentes en el servidor), la compatibilidad con entornos de hosting que pueden tener limitaciones con WebSockets, y la suficiencia para el caso de uso (el handshake de WebRTC típicamente requiere solo unos pocos mensajes de intercambio, por lo que el overhead del polling es aceptable). Cada peer hace polling cada 2-3 segundos, lo que proporciona una latencia de establecimiento de conexión de 2-6 segundos en el peor caso, aceptable para la mayoría de los casos de uso.

Decisión 2: Almacenamiento en Memoria para Señalización y Configuración El backend utiliza almacenamiento en memoria (objetos Map de JavaScript) para las salas de señalización y la configuración de Decart AI. Esto significa que los datos se pierden cuando el servidor se reinicia. Esta decisión se tomó por simplicidad y para evitar la necesidad de configurar una base de datos persistente para un prototipo funcional. Para una implementación de producción, sería recomendable reemplazar este almacenamiento por una base de datos como Redis (para las salas de señalización) o PostgreSQL (para la configuración persistente).

Decisión 3: Procesamiento de Frames por Intervalo en lugar de MediaRecorder API Para el envío de frames a Decart AI, se optó por capturar frames individuales del video usando un canvas oculto y un intervalo de tiempo, en lugar de utilizar la API MediaRecorder para capturar el stream completo. Esta decisión se basó en la necesidad de enviar frames individuales a la API de Decart AI, que espera imágenes base64 en lugar de un stream de video continuo. El intervalo de 200ms (5 FPS) fue elegido como un balance entre calidad visual y uso de recursos de la API, considerando que Decart AI cobra por créditos y un frame rate más alto consumiría créditos más rápidamente.

Decisión 4: Renderizado en Canvas en lugar de Video Element para Output Filtrado El video filtrado por Decart AI se renderiza en un elemento <canvas> en lugar de intentar modificar el stream del elemento <video>. Esto se debe a que la API de Decart AI devuelve frames procesados como imágenes base64, y el canvas es el elemento HTML más apropiado para renderizar imágenes dinámicas de manera eficiente. Además, el canvas permite un control más granular sobre el rendering y facilita futuras extensiones como la superposición de overlays o efectos adicionales.

Decisión 5: Uso de wouter en lugar de React Router Para el enrutamiento en el frontend, se utilizó wouter en lugar de la más conocida react-router. Esta decisión se basó en la simplicidad de wouter (es significativamente más ligero), su compatibilidad con el enfoque de hooks de React, y su suficiencia para las necesidades del proyecto (solo tres rutas simples). Para un proyecto con requerimientos de enrutamiento más complejos, React Router sería una elección más apropiada.

Decisión 6: Monorepo con pnpm Workspaces El proyecto está estructurado como un monorepo utilizando pnpm workspaces, lo que permite compartir código entre el frontend y el backend (como los schemas de validación de Zod generados desde OpenAPI). Esta estructura facilita el desarrollo, el testing, y el despliegue coordinado de ambas partes del sistema.

3. STACK TECNOLÓGICO
3.1 Tecnologías de Frontend
React 19: El framework principal para la construcción de la interfaz de usuario. React proporciona un modelo de componentes reutilizables, un sistema de estado eficiente con hooks, y un ecosistema maduro de bibliotecas y herramientas. La versión 19 se utiliza por sus mejoras de rendimiento y nuevas características.

TypeScript 5.9: El lenguaje de programación principal para el frontend. TypeScript añade tipado estático a JavaScript, lo que permite detectar errores en tiempo de compilación, mejora la autocompletación en el IDE, y facilita el refactoring. El proyecto utiliza una configuración TypeScript estricta para maximizar la seguridad de tipos.

Vite 7.3: La herramienta de construcción (build tool) para el frontend. Vite proporciona un servidor de desarrollo con Hot Module Replacement (HMR) extremadamente rápido, y un proceso de build optimizado que genera bundles eficientes. La configuración de Vite incluye plugins para React, Tailwind CSS, y utilidades de desarrollo de Replit.

Tailwind CSS 4: El framework de CSS utilitario para el estilizado. Tailwind permite construir interfaces modernas y responsivas sin escribir CSS personalizado, utilizando clases predefinidas que se aplican directamente en los elementos JSX. La configuración incluye un tema personalizado que se alinea con la estética del proyecto (fondo oscuro, colores de acento en púrpura y verde).

Shadcn UI: Una colección de componentes de UI reutilizables construidos sobre Radix UI primitives y estilizados con Tailwind CSS. Shadcn UI proporciona componentes accesibles, bien diseñados, y personalizables (como botones, inputs, diálogos, toasts) que aceleran significativamente el desarrollo de la interfaz.

Lucide React: La biblioteca de iconos utilizada en el proyecto. Lucide proporciona un conjunto completo de iconos vectoriales, limpios y modernos, que se utilizan para indicar estados, acciones, y categorías en la interfaz.

Wouter: El enrutador minimalista para React. Wouter proporciona enrutamiento basado en hooks con una API simple y un tamaño de bundle muy pequeño, suficiente para las tres rutas del proyecto.

TanStack Query: La biblioteca para la gestión de datos del servidor. TanStack Query (anteriormente React Query) proporciona hooks para fetching, caching, sincronización, y actualización de datos del servidor, con soporte para manejo de estados de loading, error, y success.

Framer Motion: La biblioteca para animaciones en React. Framer Motion permite crear animaciones fluidas y declarativas para transiciones de interfaz, estados de carga, y efectos visuales, mejorando la experiencia del usuario.

3.2 Tecnologías de Backend
Node.js 24: El runtime de JavaScript para el servidor. Node.js permite ejecutar JavaScript en el lado del servidor, proporcionando un modelo de eventos no bloqueantes que es ideal para aplicaciones de I/O intensivo como servidores web.

Express 5.2: El framework web para Node.js. Express proporciona una API minimalista y flexible para construir aplicaciones web, con soporte para routing, middleware, y manejo de peticiones HTTP. La versión 5 incluye mejoras de rendimiento y manejo de errores.

TypeScript 5.9: El lenguaje de programación para el backend, utilizado con las mismas ventajas que en el frontend: tipado estático, detección temprana de errores, y mejor experiencia de desarrollo.

Pino: La biblioteca de logging de alto rendimiento. Pino proporciona logging estructurado en formato JSON, que es ideal para aplicaciones en producción donde los logs deben ser parseables por herramientas de análisis. En desarrollo, se utiliza pino-pretty para formatear los logs de manera legible.

CORS: El middleware para habilitar Cross-Origin Resource Sharing. CORS permite que el frontend (que se ejecuta en un dominio/puerto diferente al del backend) pueda hacer peticiones al backend sin restricciones de seguridad del navegador.

Zod: La biblioteca de validación de schemas. Zod se utiliza para validar los datos de entrada en las peticiones HTTP, asegurando que los datos recibidos cumplan con los formatos esperados antes de ser procesados. Los schemas de Zod se generan automáticamente desde la especificación OpenAPI.

ESBuild: La herramienta de bundling para el backend. ESBuild se utiliza para compilar el código TypeScript del backend en un bundle JavaScript optimizado, con soporte para ESM (ECMAScript Modules) y source maps para debugging.

3.3 Tecnologías de Infraestructura y Desarrollo
pnpm: El gestor de paquetes y workspaces. pnpm proporciona un sistema de workspaces que permite gestionar múltiples paquetes en un solo repositorio, compartiendo dependencias y facilitando el desarrollo coordinado.

OpenAPI 3.1: La especificación para la API RESTful. La especificación OpenAPI define todos los endpoints, parámetros, schemas, y respuestas de la API, sirviendo como contrato entre el frontend y el backend. Se utiliza Orval para generar hooks de React Query y schemas de Zod a partir de esta especificación.

Replit: La plataforma de desarrollo en la nube. Replit proporciona el entorno de desarrollo, el hosting de los servicios (mediante workflows), y el sistema de variables de entorno para el proyecto.

4.2 Punto de Entrada (main.tsx)
El archivo main.tsx es el punto de entrada de la aplicación React. Su función es renderizar el componente raíz App en el elemento DOM con id root. Este archivo típicamente incluye las importaciones necesarias para React, ReactDOM, y los estilos globales.

4.3 Componente Raíz (App.tsx)
El archivo App.tsx es el componente raíz que encapsula toda la aplicación. Su responsabilidad principal es configurar los proveedores de contexto necesarios y definir el sistema de rutas. El componente App devuelve una estructura JSX que incluye:

QueryClientProvider: Proporciona el cliente de TanStack Query a toda la aplicación, permitiendo que cualquier componente hijo utilice los hooks de fetching de datos.
TooltipProvider: Proporciona el contexto para los tooltips de Radix UI, asegurando que los tooltips funcionen correctamente en toda la aplicación.
WouterRouter: El componente de enrutamiento que define la base de las rutas y permite la navegación entre páginas.
Router: El componente interno que define las rutas específicas y sus componentes asociados.
Toaster: El componente que renderiza las notificaciones toast en la aplicación.
Las rutas definidas son:

/ (raíz): Renderiza el componente PcReceiver, que es la página principal del PC.
/mobile: Renderiza el componente MobileSender, que es la página del dispositivo móvil.
/config: Renderiza el componente ConfigPage, que es la página de configuración de Decart AI.
Ruta por defecto: Renderiza el componente NotFound para cualquier ruta no reconocida.

4.4.3 Funciones Principales
toggleFullscreen: Gestiona la entrada y salida del modo de pantalla completa utilizando la API de Fullscreen del navegador (requestFullscreen y exitFullscreen). Actualiza el estado isFullscreen para reflejar el cambio visual en la interfaz.

captureFrame: Captura un frame del video actual dibujándolo en un canvas oculto y convirtiéndolo a una cadena base64 en formato JPEG. Esta función es crucial para el pipeline de procesamiento con Decart AI. La calidad de compresión JPEG se fija en 0.85 (85%) como un balance entre calidad visual y tamaño de datos.

connectToDecart: Establece la conexión WebSocket con el servicio de Decart AI. Envía el mensaje de configuración inicial con la API key, el prompt, y otros parámetros. Gestiona los eventos del WebSocket: onopen (conexión establecida), onmessage (recepción de frames filtrados o mensajes de error), onerror (errores de conexión), y onclose (cierre de conexión). Inicia un loop de renderizado mediante requestAnimationFrame para mostrar los frames filtrados en el canvas.

startFrameProcessing: Inicia un intervalo que captura frames del video y los envía al WebSocket de Decart AI. Implementa control de backpressure mediante isProcessingRef para evitar el envío de frames si el frame anterior aún no ha sido procesado. La frecuencia de 5 FPS (200ms entre frames) fue elegida para balancear calidad y uso de créditos de la API.

startRenderLoop: Inicia un loop de renderizado continuo que dibuja los frames filtrados recibidos de Decart AI en el canvas visible. Utiliza requestAnimationFrame para sincronizar el renderizado con el refresh rate del monitor.

startReceiving: La función principal que inicia todo el proceso de recepción. Crea una sala de señalización si no existe, configura la conexión WebRTC, inicia el polling para la oferta del móvil, y gestiona los eventos de la conexión peer-to-peer.

stopReceiving: Limpia todos los recursos y estados cuando el usuario decide detener la conexión. Cierra la conexión WebRTC, el WebSocket de Decart AI, los intervalos de polling y captura de frames, y resetea todos los estados de la interfaz.

6.2 Punto de Entrada (index.ts)
El archivo index.ts es el punto de entrada del servidor. Su función es importar la aplicación Express configurada, obtener el puerto de la variable de entorno PORT, y iniciar el servidor HTTP. El archivo incluye validación estricta de la variable de entorno, lanzando un error si no está definida o si el valor no es un número válido. El servidor se inicia con app.listen(port, callback), y el callback registra un mensaje de log con el puerto en el que el servidor está escuchando.

6.3 Configuración de Express (app.ts)
El archivo app.ts configura la aplicación Express con los siguientes middlewares, en orden:

pinoHttp: Middleware de logging que registra todas las peticiones HTTP. Se configura con el logger de Pino y serializadores personalizados que extraen solo la información relevante (ID de la petición, método HTTP, URL, y código de estado de la respuesta). Los headers de autorización y cookies se redactan por seguridad.

cors: Middleware que habilita CORS para todas las rutas, permitiendo que el frontend (que se ejecuta en un dominio diferente) pueda hacer peticiones al backend.

express.json: Middleware que parsea el body de las peticiones con Content-Type application/json, convirtiendo el JSON en un objeto JavaScript accesible en req.body.

express.urlencoded: Middleware que parsea el body de las peticiones con Content-Type application/x-www-form-urlencoded, permitiendo el procesamiento de formularios HTML.

Router de /api: El router principal que monta todas las sub-rutas del backend bajo el prefijo /api.

6.4 Router Principal (routes/index.ts)
El archivo routes/index.ts es el router principal que agrega todos los sub-routers:

healthRouter: Montado en /healthz (ruta completa: /api/healthz).
signalingRouter: Montado en /signaling (ruta completa: /api/signaling/*).
decartRouter: Montado en /decart (ruta completa: /api/decart/*).
6.5 Endpoint de Salud (routes/health.ts)
El endpoint GET /api/healthz proporciona un simple health check que devuelve { status: "ok" }. Utiliza el schema de Zod HealthCheckResponse (generado desde OpenAPI) para validar la respuesta, asegurando que cumpla con el contrato de la API.

6.6 Endpoints de Señalización (routes/signaling.ts)
Este archivo contiene la implementación completa del servidor de señalización WebRTC. Utiliza un objeto Map de JavaScript para almacenar las salas en memoria. Cada sala tiene la siguiente estructura:

{
  id: string;              // ID único de la sala
  name: string;            // Nombre descriptivo
  status: string;          // Estado: "waiting", "connected", "closed"
  createdAt: string;       // Fecha de creación en formato ISO
  mobileConnected: boolean; // Indica si el móvil ha enviado una oferta
  pcConnected: boolean;    // Indica si el PC ha enviado una respuesta
  offer: any | null;       // La oferta SDP del móvil
  answer: any | null;      // La respuesta SDP del PC
  mobileIce: any[];        // Candidatos ICE del móvil
  pcIce: any[];            // Candidatos ICE del PC
}

POST /api/signaling/rooms Crea una nueva sala. Genera un ID único usando Date.now().toString(36) concatenado con un fragmento aleatorio de Math.random().toString(36).substr(2, 5). Esto produce un ID de 8-9 caracteres alfanuméricos que es suficientemente único para el caso de uso. Inicializa la sala con todos los campos en su estado por defecto y devuelve el objeto de la sala con código de estado 201.

GET /api/signaling/rooms/:roomId Obtiene los detalles de una sala existente. Si la sala no existe, devuelve 404. Devuelve un objeto con el ID, estado, fecha de creación, y estados de conexión de ambos pares.

POST /api/signaling/rooms/:roomId/offer Almacena la oferta SDP del móvil. Establece mobileConnected en true y, si el PC ya está conectado, cambia el estado de la sala a "connected". Devuelve un mensaje de éxito.

GET /api/signaling/rooms/:roomId/offer Obtiene la oferta SDP del móvil. Si no hay oferta, devuelve { success: false, message: "No offer yet" }. Si hay oferta, devuelve { success: true, message: "Offer available", ...offer }.

POST /api/signaling/rooms/:roomId/answer Almacena la respuesta SDP del PC. Establece pcConnected en true y, si el móvil ya está conectado, cambia el estado de la sala a "connected". Devuelve un mensaje de éxito.

GET /api/signaling/rooms/:roomId/answer Obtiene la respuesta SDP del PC. Si no hay respuesta, devuelve { success: false, message: "No answer yet" }. Si hay respuesta, devuelve { success: true, message: "Answer available", ...answer }.

POST /api/signaling/rooms/:roomId/ice Almacena un candidato ICE. El body de la petición debe incluir candidate, sdpMid, sdpMLineIndex, y source ("mobile" o "pc"). Según el valor de source, el candidato se añade al array mobileIce o pcIce de la sala.

GET /api/signaling/rooms/:roomId/ice/mobile Obtiene todos los candidatos ICE del móvil. Devuelve { candidates: room.mobileIce }.

GET /api/signaling/rooms/:roomId/ice/pc Obtiene todos los candidatos ICE del PC. Devuelve { candidates: room.pcIce }.

6.7 Endpoints de Configuración Decart (routes/decart.ts)
Este archivo gestiona la configuración de la integración con Decart AI. La configuración se almacena en una variable global en memoria.

GET /api/decart/config Devuelve la configuración actual, con la API key enmascarada (mostrando "***" si existe, o cadena vacía si no existe). Esto permite que el frontend verifique si hay una configuración guardada sin exponer la API key real.

PUT /api/decart/config Actualiza la configuración. Acepta un body JSON con los campos apiKey, model, mirror, enhance, y prompt. Solo actualiza los campos que están presentes en el body, permitiendo actualizaciones parciales. Devuelve la configuración actualizada con la API key enmascarada.

6.8 Logger (lib/logger.ts)
El archivo lib/logger.ts configura el logger de Pino. La configuración incluye:

Nivel de log: Se obtiene de la variable de entorno LOG_LEVEL o se usa "info" por defecto.
Redactación: Se redactan los headers authorization y cookie de las peticiones, y los headers set-cookie de las respuestas, para evitar la filtración de información sensible en los logs.
Transporte: En desarrollo (NODE_ENV !== "production"), se utiliza pino-pretty para formatear los logs con colores. En producción, los logs se emiten en formato JSON estructurado para facilitar el parsing por herramientas de log aggregation.

PARTE V: PROTOCOLO DE SEÑALIZACIÓN WEBRTC
8. FLUJO DE SEÑALIZACIÓN DETALLADO
8.1 Fase de Inicialización
La fase de inicialización comienza cuando el usuario decide iniciar una sesión de transmisión. El proceso es asimétrico: el PC (receptor) debe estar listo antes de que el móvil (emisor) comience la transmisión.

Paso 1: Creación de la Sala por el PC El PC envía una petición POST a /api/signaling/rooms con un body JSON que contiene el nombre de la sala. El servidor genera un ID único, crea la sala en memoria, y devuelve el objeto de la sala con código 201. El PC almacena el roomId y lo muestra al usuario.

Paso 2: Inicio del Polling de Oferta por el PC El PC inicia un intervalo que consulta cada 2 segundos el endpoint /api/signaling/rooms/{roomId}/offer. Inicialmente, la respuesta será { success: false, message: "No offer yet" } porque el móvil aún no ha enviado la oferta. El PC continúa haciendo polling indefinidamente hasta que recibe una oferta válida o el usuario cancela la operación.

Paso 3: Inicio de la Transmisión por el Móvil El usuario abre la página /mobile en su dispositivo y hace clic en "Iniciar transmisión". El móvil solicita permisos de cámara y micrófono mediante la API navigator.mediaDevices.getUserMedia. La configuración solicitada es { video: { width: 1280, height: 720, facingMode: "user" }, audio: true }, que solicita video HD desde la cámara frontal y audio.

Paso 4: Creación de la Conexión WebRTC y Oferta por el Móvil El móvil crea un objeto RTCPeerConnection con la configuración de servidores ICE de Google:

{ iceServers: [{ urls: "stun:stun.l.google.com:19302" }, { urls: "stun:stun1.l.google.com:19302" }] }

Añade los tracks del stream de cámara a la conexión mediante pc.addTrack(track, stream). Configura el callback onicecandidate para recolectar candidatos ICE. Genera la oferta mediante pc.createOffer() y la establece como descripción local mediante pc.setLocalDescription(offer).

Paso 5: Envío de la Oferta por el Móvil El móvil envía la oferta al backend mediante POST a /api/signaling/rooms/{roomId}/offer con el body:

{ "type": "offer", "sdp": "v=0\no=- 1234567890 2 IN IP4 127.0.0.1\ns=-\nt=0 0\n..." }

El servidor almacena la oferta en la sala y establece mobileConnected en true.

Paso 6: Recolección de Candidatos ICE por el Móvil El móvil espera 2 segundos para permitir que se recolecten candidatos ICE, y luego envía todos los candidatos recolectados al backend mediante POST a /api/signaling/rooms/{roomId}/ice con el body:

{ "candidate": "candidate:1234567890 1 udp 1234567890 192.168.1.100 12345 typ host", "sdpMid": "0", "sdpMLineIndex": 0, "source": "mobile" }

Paso 7: Inicio del Polling de Respuesta por el Móvil El móvil inicia un intervalo que consulta cada 2 segundos el endpoint /api/signaling/rooms/{roomId}/answer. Inicialmente, la respuesta será { success: false, message: "No answer yet" }.

Paso 8: Inicio del Polling de Candidatos ICE por el Móvil El móvil inicia un intervalo que consulta cada 3 segundos el endpoint /api/signaling/rooms/{roomId}/ice/pc para obtener los candidatos ICE del PC. Añade cada candidato recibido a la conexión WebRTC mediante pc.addIceCandidate(candidate).

8.2 Fase de Establecimiento
Paso 9: Detección de la Oferta por el PC El PC recibe la oferta del móvil en una de sus consultas de polling. La respuesta ahora incluye { success: true, message: "Offer available", type: "offer", sdp: "..." }. El PC establece la oferta como descripción remota mediante pc.setRemoteDescription(new RTCSessionDescription({ type: "offer", sdp: data.sdp })).

Paso 10: Creación de la Respuesta por el PC El PC crea una respuesta mediante pc.createAnswer() y la establece como descripción local mediante pc.setLocalDescription(answer). Configura el callback ontrack para recibir el stream de video del móvil. Configura el callback onicecandidate para recolectar candidatos ICE.

Paso 11: Envío de la Respuesta por el PC El PC envía la respuesta al backend mediante POST a /api/signaling/rooms/{roomId}/answer con el body:

{ "type": "answer", "sdp": "v=0\no=- 9876543210 2 IN IP4 127.0.0.1\ns=-\nt=0 0\n..." }

El servidor almacena la respuesta en la sala y establece pcConnected en true. Si mobileConnected también es true, el estado de la sala cambia a "connected".

Paso 12: Recolección de Candidatos ICE por el PC El PC espera 2 segundos para permitir que se recolecten candidatos ICE, y luego envía todos los candidatos recolectados al backend mediante POST a /api/signaling/rooms/{roomId}/ice con source: "pc".

Paso 13: Inicio del Polling de Candidatos ICE por el PC El PC inicia un intervalo que consulta cada 3 segundos el endpoint /api/signaling/rooms/{roomId}/ice/mobile para obtener los candidatos ICE del móvil. Añade cada candidato recibido a la conexión WebRTC.

Paso 14: Detección de la Respuesta por el Móvil El móvil recibe la respuesta del PC en una de sus consultas de polling. La respuesta ahora incluye { success: true, message: "Answer available", type: "answer", sdp: "..." }. El móvil establece la respuesta como descripción remota mediante pc.setRemoteDescription(new RTCSessionDescription(data)).

8.3 Fase de Conexión Directa
Paso 15: Intercambio de Candidatos ICE Ambos pares continúan intercambiando candidatos ICE a través del backend. Cada candidato recibido se añade a la conexión WebRTC mediante addIceCandidate. Los candidatos pueden incluir direcciones locales (host), direcciones NAT (srflx), y direcciones de relay (relay) si se configura un servidor TURN.

Paso 16: Establecimiento de la Conexión Peer-to-Peer Una vez que ambos pares tienen las descripciones de sesión y suficientes candidatos ICE, el agente de ICE de WebRTC intenta establecer una conexión directa entre los pares. El proceso de ICE prueba diferentes combinaciones de candidatos (local-local, local-NAT, NAT-NAT, etc.) hasta encontrar una ruta que funcione. Si se encuentra una ruta, el estado de la conexión cambia a "connected" y el flujo de datos comienza.

Paso 17: Inicio del Flujo de Video El evento ontrack se dispara en el PC cuando el primer track de media llega a través de la conexión. El PC asigna el stream recibido al elemento <video> mediante videoRef.current.srcObject = event.streams[0]. El video comienza a reproducirse automáticamente (gracias al atributo autoPlay).

8.4 Fase de Procesamiento con Decart AI
Paso 18: Conexión WebSocket con Decart AI Si el usuario ha configurado una API key de Decart AI, el PC establece una conexión WebSocket con wss://api3.decart.ai/v1/stream?model=lucy-2.1. El parámetro de query model especifica el modelo de IA a utilizar.

Paso 19: Configuración de Decart AI Una vez que el WebSocket se abre, el PC envía un mensaje de configuración:

{
  "type": "config",
  "apiKey": "sk-...",
  "prompt": "Anime style portrait",
  "enhance": true,
  "mirror": "auto"
}

El servidor de Decart AI valida la API key y devuelve un mensaje de confirmación (config_ack).

Paso 20: Captura de Frames El PC inicia un intervalo que ejecuta cada 200ms. En cada ejecución, captura un frame del video actual dibujándolo en un canvas oculto y convirtiéndolo a base64:

const canvas = document.createElement('canvas');
canvas.width = video.videoWidth;
canvas.height = video.videoHeight;
const ctx = canvas.getContext('2d');
ctx.drawImage(video, 0, 0);
const frameData = canvas.toDataURL('image/jpeg', 0.85);

Paso 21: Envío de Frames a Decart AI El PC envía el frame capturado al WebSocket:

{ "type": "frame", "frame": "data:image/jpeg;base64,/9j/4AAQSkZJRgABAQAAAQ..." }

El sistema implementa backpressure: si el frame anterior aún no ha sido procesado (isProcessingRef.current === true), no se envía un nuevo frame. Esto evita la saturación de la API y el consumo innecesario de créditos.

Paso 22: Recepción de Frames Procesados Decart AI procesa el frame y devuelve el resultado como otro mensaje de tipo "frame":

{ "type": "frame", "frame": "data:image/jpeg;base64,/9j/4AAQSkZJRgABAQAAAQ..." }

El PC almacena el frame procesado en lastFrameRef.current y marca isProcessingRef.current como false, permitiendo que el siguiente frame se envíe.

Paso 23: Renderizado de Frames Procesados El PC renderiza el frame procesado en un canvas visible utilizando un loop de requestAnimationFrame. El canvas se muestra en lugar del video original cuando el usuario ha seleccionado el modo "Filtrado".

8.5 Fase de Finalización
Paso 24: Detención por el Usuario Cuando el usuario hace clic en "Detener" en cualquiera de los dispositivos, se ejecuta la secuencia de limpieza:

En el PC:

Cierra la conexión WebRTC (pc.close())
Cierra el WebSocket de Decart AI (ws.close())
Detiene los intervalos de polling (clearInterval)
Detiene el intervalo de captura de frames (clearInterval)
Resetea el estado de la interfaz (setIsConnected(false), etc.)
Limpia el srcObject del video (videoRef.current.srcObject = null)
En el móvil:

Cierra la conexión WebRTC (pc.close())
Detiene los tracks de media (stream.getTracks().forEach(t => t.stop()))
Detiene los intervalos de polling (clearInterval)
Resetea el estado de la interfaz
Paso 25: Limpieza del Servidor El servidor actualmente no realiza limpieza automática de salas inactivas. En una implementación de producción, se debería implementar un mecanismo de TTL (Time To Live) que elimine las salas después de un período de inactividad (por ejemplo, 1 hora) para evitar la acumulación de salas en memoria.

PARTE VI: INTEGRACIÓN CON DECART AI
9. PROTOCOLO DE COMUNICACIÓN CON DECART AI
9.1 Endpoint WebSocket
La conexión con Decart AI se establece mediante WebSocket en la URL:

wss://api3.decart.ai/v1/stream?model=lucy-2.1

El parámetro model especifica el modelo de IA a utilizar. Actualmente solo se soporta el modelo lucy-2.1, que es un modelo de generación de imágenes optimizado para la transformación de retratos y video en tiempo real.

9.2 Mensajes del Cliente al Servidor
Mensaje de Configuración (tipo: config) Debe enviarse inmediatamente después de que la conexión WebSocket se abra. Contiene la configuración necesaria para que el servidor de Decart AI sepa cómo procesar los frames.

{
  "type": "config",
  "apiKey": "string",        // La API key de Decart AI
  "prompt": "string",        // Descripción del estilo/filtro deseado
  "enhance": boolean,        // Indica si se debe mejorar la calidad
  "mirror": "string"         // Configuración de espejo: "auto", "true", "false"
}

Mensaje de Frame (tipo: frame) Se envía para cada frame que se desea procesar. El frame debe ser una imagen codificada en base64 con el prefijo MIME data:image/jpeg;base64,.

{
  "type": "frame",
  "frame": "data:image/jpeg;base64,/9j/4AAQSkZJRgABAQAAAQ..."
}

9.3 Mensajes del Servidor al Cliente
Mensaje de Confirmación de Configuración (tipo: config_ack) Indica que la configuración fue recibida y aceptada. El servidor está listo para recibir frames.

{
  "type": "config_ack"
}

Mensaje de Frame Procesado (tipo: frame) Contiene el frame procesado por la IA. El frame está codificado en base64 con el prefijo MIME.

{
  "type": "frame",
  "frame": "data:image/jpeg;base64,/9j/4AAQSkZJRgABAQAAAQ..."
}

Mensaje de Error (tipo: error) Indica que ocurrió un error durante el procesamiento.

{
  "type": "error",
  "message": "string"  // Descripción del error
}

9.4 Control de Backpressure
El sistema implementa control de backpressure para evitar que se envíen frames más rápido de lo que el servidor puede procesarlos. La lógica es la siguiente:

Se inicializa una variable isProcessing en false.
Cuando se envía un frame, isProcessing se establece en true.
Solo se envía un nuevo frame si isProcessing es false.
Cuando se recibe la respuesta del servidor (ya sea un frame procesado o un error), isProcessing se establece en false.
Esta lógica garantiza que nunca haya más de un frame en vuelo en un momento dado, lo que previene la saturación de la red y el consumo excesivo de créditos de la API.

9.5 Frecuencia de Captura de Frames
La frecuencia de captura de frames está configurada en 5 FPS (frames por segundo), lo que corresponde a un intervalo de 200ms entre frames. Esta frecuencia fue elegida como un balance entre:

Calidad visual: 5 FPS es suficiente para mostrar el efecto del filtro de IA en tiempo real, aunque el movimiento puede parecer ligeramente entrecortado.
Costo de la API: Decart AI cobra por créditos. A 5 FPS, el costo es aproximadamente 2 créditos por segundo, que es razonable para la mayoría de los usuarios.
Rendimiento del navegador: Capturar, codificar, y enviar frames a 5 FPS no sobrecarga significativamente el navegador del PC.
En una implementación futura, se podría hacer configurable la frecuencia de captura, permitiendo al usuario elegir entre 1 FPS (muy económico), 5 FPS (balance), 10 FPS (más fluido), o 30 FPS (máxima calidad, pero muy costoso).

9.6 Calidad de Compresión JPEG
Los frames se comprimen en formato JPEG con una calidad del 85% (0.85 en el parámetro de toDataURL). Esta calidad fue elegida como un balance entre:

Calidad visual: 85% preserva la mayoría de los detalles importantes de la imagen.
Tamaño de datos: 85% produce archivos JPEG de tamaño moderado, reduciendo el tiempo de transmisión y el uso de ancho de banda.
Velocidad de procesamiento: Imágenes más pequeñas se procesan más rápidamente en el servidor de Decart AI.
9.7 Resolución de Frames
Los frames se capturan a la resolución nativa del video WebRTC, que típicamente es 1280x720 (HD) basado en la configuración de getUserMedia en el móvil. Si se desea reducir el tamaño de los frames para acelerar el procesamiento o reducir el costo, se podría implementar un redimensionamiento previo antes de enviar los frames a Decart AI. Por ejemplo, redimensionar a 640x360 reduciría el tiempo de procesamiento aproximadamente a la mitad y también reduciría el costo.

PARTE VIII: INSTRUCCIONES DE USO
11. GUÍA DE USUARIO
11.1 Primeros Pasos
Paso 1: Acceder a la Aplicación Abra su navegador web y acceda a la URL principal de la aplicación (la URL raíz /). Esta página es el receptor de PC y debe abrirse en la computadora que recibirá el video y aplicará los filtros.

Paso 2: Iniciar la Recepción en el PC En la página principal del PC, haga clic en el botón "Iniciar recepción". Esto creará una sala de señalización y mostrará un ID de sala único. El PC comenzará a esperar la conexión del móvil.

Paso 3: Abrir la Página del Móvil En su dispositivo móvil, abra la página /mobile (añadiendo /mobile a la URL principal). Esta página está optimizada para dispositivos móviles y proporciona la interfaz para capturar la cámara.

Paso 4: Iniciar la Transmisión desde el Móvil En la página del móvil, haga clic en "Iniciar transmisión". El navegador solicitará permiso para acceder a la cámara y al micrófono. Autorice el acceso. El móvil creará automáticamente una conexión WebRTC y comenzará a transmitir el video.

Paso 5: Verificar la Conexión En el PC, debería ver el video de la cámara del móvil aparecer en la pantalla. Un indicador "En vivo" confirmará que la conexión está activa. El proceso de conexión típicamente toma 2-6 segundos.

11.2 Configuración de Decart AI
Paso 1: Obtener una API Key Visite https://decart.ai y cree una cuenta. Obtenga una API key desde el panel de control de su cuenta. Decart AI opera con un sistema de créditos: cada segundo de procesamiento de video consume aproximadamente 2 créditos.

Paso 2: Configurar la API Key en el PC En la página principal del PC, haga clic en el icono de engranaje (⚙️) para abrir el panel de configuración. Introduzca su API key en el campo "Decart AI API Key". El campo es de tipo password para proteger la visibilidad de la key.

Paso 3: Configurar el Prompt de Filtro En el mismo panel de configuración, introduzca un prompt que describa cómo desea que se vea el video filtrado. Algunos ejemplos de prompts:

"Anime style portrait"
"Oil painting style"
"Watercolor portrait"
"Cyberpunk neon style"
"Van Gogh painting style"
"3D cartoon character"
"Renaissance oil painting"
"Pop art style"
Paso 4: Guardar la Configuración Haga clic en "Guardar configuración". La configuración se almacenará en el servidor y se aplicará automáticamente a la próxima transmisión.

Paso 5: Activar los Filtros Una vez que el video está siendo recibido y la API key está configurada, los filtros se activarán automáticamente. Verá un indicador "Filtro Decart AI activo" en la esquina superior izquierda. Puede alternar entre el video original y el filtrado usando el botón "Original/Filtrado" en la esquina inferior derecha.

11.3 Uso de SplitCam
Paso 1: Instalar SplitCam Descargue e instale SplitCam desde https://splitcam.com (disponible para Windows).

Paso 2: Configurar la Fuente de Video En SplitCam, seleccione "Window capture" o "Browser capture" como fuente de video. Seleccione la ventana del navegador donde está abierta la aplicación Decart AI Camera.

Paso 3: Ajustar la Región de Captura Ajuste la región de captura para que incluya solo el área de video, excluyendo los controles de la interfaz. Puede usar el modo fullscreen de la aplicación para facilitar esto.

Paso 4: Activar la Cámara Virtual En SplitCam, active la cámara virtual. Esto creará una nueva cámara virtual que otras aplicaciones pueden utilizar.

Paso 5: Usar en WhatsApp u Otras Aplicaciones Abra WhatsApp Web o la aplicación de escritorio de WhatsApp. En la configuración de video, seleccione la cámara virtual de SplitCam. Ahora sus contactos de WhatsApp verán el video filtrado de su cámara móvil.

11.4 Modo Fullscreen
Para una experiencia inmersiva y facilitar la captura por parte de software de terceros, puede activar el modo fullscreen:

Haga clic en el botón de pantalla completa (icono de cuatro flechas) en la esquina inferior derecha.
O utilice el atajo de teclado F11 del navegador.
En modo fullscreen, todos los controles de la interfaz se ocultan automáticamente, mostrando solo el video.
Para salir del modo fullscreen, presione Escape o haga clic en el botón de minimizar.
11.5 Detener la Transmisión
Para detener la transmisión en cualquier momento:

En el PC, haga clic en "Detener".
En el móvil, haga clic en "Detener".
La conexión WebRTC se cerrará, el video dejará de transmitirse, y todos los recursos se liberarán.
PARTE IX: TROUBLESHOOTING
12. GUÍA DE RESOLUCIÓN DE PROBLEMAS
12.1 Problemas de Conexión WebRTC
Síntoma: El PC no recibe el video del móvil

Verifique que ambos dispositivos están en la misma red: WebRTC funciona mejor cuando ambos dispositivos están en la misma red local. Si están en redes diferentes, puede ser necesario configurar un servidor TURN.

Verifique que los servidores STUN están configurados: El sistema utiliza los servidores STUN de Google. Asegúrese de que estos servidores sean accesibles desde ambos dispositivos (no estén bloqueados por firewalls).

Verifique el ID de la sala: Asegúrese de que el móvil esté usando el mismo ID de sala que el PC. El ID se muestra en ambas interfaces.

Verifique los permisos de cámara: En el móvil, asegúrese de que el navegador tiene permiso para acceder a la cámara. Puede verificar esto en la configuración de permisos del navegador.

Verifique la consola del navegador: Abra la consola del navegador (F12) en ambos dispositivos y busque mensajes de error relacionados con WebRTC.

Intente en diferentes navegadores: WebRTC tiene soporte variable entre navegadores. Chrome y Firefox generalmente tienen el mejor soporte.

12.2 Problemas con Decart AI
Síntoma: Los filtros no se aplican

Verifique la API key: Asegúrese de que la API key está correctamente configurada en el panel de configuración. La key debe ser válida y tener créditos disponibles.

Verifique la conexión WebSocket: Abra la consola del navegador y busque mensajes relacionados con el WebSocket. Debería ver mensajes de conexión, configuración, y frames.

Verifique el prompt: Asegúrese de que el prompt no está vacío y es descriptivo. Un prompt muy corto o genérico puede no producir resultados visibles.

Verifique los créditos: Visite https://decart.ai y verifique que su cuenta tiene créditos disponibles. Si los créditos se agotaron, los filtros dejarán de aplicarse.

Verifique el estado en la interfaz: El indicador de estado en la esquina superior izquierda muestra el estado actual de la conexión con Decart AI. Si dice "Video directo", los filtros no están activos.

12.3 Problemas de Rendimiento
Síntoma: El video se retrasa o se congela

Reduzca la resolución: En el móvil, la configuración de getUserMedia solicita 1280x720. Puede reducir esto a 640x480 para reducir el ancho de banda necesario.

Reduzca la frecuencia de frames: Si está usando Decart AI, reducir la frecuencia de captura de frames (de 5 FPS a 1-2 FPS) reducirá la carga en el servidor y el consumo de créditos.

Cierre otras aplicaciones: Asegúrese de que no hay otras aplicaciones intensivas en CPU o red ejecutándose en el PC.

Use una conexión cableada: Si es posible, conecte el PC a la red mediante Ethernet en lugar de Wi-Fi para reducir la latencia.

12.4 Problemas de SplitCam
Síntoma: SplitCam no detecta la ventana del navegador

Ejecute SplitCam como administrador: En Windows, ejecute SplitCam con privilegios de administrador para que pueda capturar ventanas de otros aplicaciones.

Use la captura de pantalla completa: En lugar de capturar una ventana específica, use la captura de pantalla completa y recorte la región del video.

Actualice SplitCam: Asegúrese de que está usando la última versión de SplitCam.

Verifique la cámara virtual: En la configuración de Windows, verifique que la cámara virtual de SplitCam está instalada y funcionando.

PARTE XII: CONCLUSIÓN
15. RESUMEN EJECUTIVO
El proyecto Decart AI Camera representa una solución completa y funcional para el problema de transmitir video de cámara móvil a PC, aplicar filtros de inteligencia artificial en tiempo real, y presentar el resultado en una ventana capturable por software de terceros. La arquitectura del sistema, basada en WebRTC para la comunicación peer-to-peer y en una API RESTful para la señalización, proporciona una base sólida y extensible.

El sistema está diseñado con los siguientes principios fundamentales:

Simplicidad: La interfaz de usuario es intuitiva y guía al usuario a través de cada paso.
Modularidad: La arquitectura de monorepo con pnpm workspaces permite un desarrollo organizado y la reutilización de código.
Extensibilidad: La integración con Decart AI está diseñada para ser reemplazable por otros servicios de IA en el futuro.
Transparencia: La especificación OpenAPI documenta exhaustivamente la API, y el código está completamente tipado con TypeScript.
Seguridad: Las API keys se manejan de forma segura (no se exponen en logs, se enmascaran en respuestas de la API).
El proyecto está en un estado funcional y puede ser utilizado inmediatamente para transmisiones de video con filtros de IA. Las futuras extensiones pueden incluir persistencia de datos, autenticación de usuarios, soporte para múltiples modelos de IA, y una aplicación nativa móvil.
